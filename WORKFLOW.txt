================================================================
  입트영 English Habit — 개발 워크플로 (한 사이클 회고)
  2026-06-21 작성
================================================================

이 문서는 "1인용 한국어→영어 학습 PWA를 0에서부터 배포까지"
만드는 동안 실제로 거친 순서를 정리한 것이다. 각 단계마다:
  [무엇을] 어떤 작업을 했는가
  [왜]    어떤 결정·이유로 그렇게 했는가
  [응용]  다른 프로젝트에 똑같이 적용할 때 핵심 포인트
를 함께 적어 둔다.

================================================================
  STEP 0. 프로젝트 골격 (Monorepo + 도구 선택)
================================================================

[무엇을]
  - 루트에 /client 와 /server 두 폴더를 두는 단일 저장소
  - /client : Vite + React 18 + TypeScript + Tailwind
  - /server : Node + Express + TypeScript + Anthropic SDK
  - 루트 package.json 에 install:all / dev / build 등 통합 스크립트
  - server/.env 에 ANTHROPIC_API_KEY 보관, .gitignore 로 차단

[왜]
  - 1인 프로젝트라 멀티 리포·CI/CD 분리는 과함. 단일 저장소가
    버전·이슈·README 한 곳에 묶여 관리가 쉽다.
  - 클라/서버를 디렉터리 단위로 분리해 두면 나중에 둘 중 하나만
    별도 배포(Render / Vercel)하기 쉽다.

[응용]
  - 풀스택 1인 프로젝트의 기본 스켈레톤으로 그대로 재사용 가능.
  - "통합 dev 명령(npm run dev)"이 있으면 매일 켜는 비용이 줄어
    실제로 작업이 지속된다. npm-run-all --parallel 가 핵심.

================================================================
  STEP 1. 데스크톱 .exe → 모바일 PWA 로 전환
================================================================

[무엇을]
  - 초기엔 Electron(packager)으로 포터블 .exe 빌드를 시도.
  - 빌드는 됐지만 실행 시 안 뜸 → 이슈 디버깅 중에 사용자가
    "어차피 폰에서 쓸 거다"고 방향 전환을 결정.
  - Electron 자취(/electron, /dist-electron, deps)를 모두 제거.
  - client 에 vite-plugin-pwa 를 도입해 manifest + sw.js 자동 생성.
  - iOS / Android 홈 화면 메타태그, 아이콘 4종(192/512/512-maskable/
    apple-touch-icon) 준비.

[왜]
  - 폰에서 쓸 거라면 Electron .exe 는 단순한 wrapper일 뿐.
  - PWA 는 같은 코드로 데스크톱·iOS·안드로이드 모두 커버되고
    "홈 화면에 추가" 만으로 풀스크린 앱처럼 동작.

[응용]
  - "이거 결국 어디서 쓸 거지?" 를 가장 먼저 결정해라. 호스트가
    바뀌면 그동안 들인 토이 코드는 거의 다 버려진다.
  - 모바일이 1순위면 처음부터 PWA 로 가는 게 정답.

================================================================
  STEP 2. 폰에서 LAN으로 dev 서버 보기 → HTTPS 터널 필요성 발견
================================================================

[무엇을]
  - vite.config 에 server.host: true 추가, 폰을 같은 Wi-Fi 로
    `http://172.30.1.44:5173` 에 접속하려 시도.
  - Windows Defender 방화벽 + Wi-Fi 가 "Public" 프로필 + GPO 락 →
    LAN 인바운드 차단으로 접속 자체가 안 됨.
  - cloudflared "quick tunnel" 로 즉시 https 임시 URL 발급.
  - vite.config 에 server.allowedHosts: true 추가해 임의 도메인 허용.

[왜]
  - LAN 의 http URL 은 보안 컨텍스트 아님 → 마이크·MediaRecorder·
    Service Worker 등이 폰 브라우저에서 거부됨.
  - cloudflared 는 https URL 을 무료·무가입으로 만들어 줘서 모바일
    검증을 한 번에 처리.

[응용]
  - 모바일에서 dev 서버 검증할 일이 한 번이라도 있다면 무조건
    HTTPS 터널을 디폴트로 깔고 시작. ngrok / cloudflared 모두 가능.
  - Vite 5.4+ 의 allowedHosts: true 옵션 잊지 말 것 (post-CVE host
    체크가 임의 도메인을 거부함).

================================================================
  STEP 3. Phase 1 — 사진 → 핵심 표현 자동 추출
================================================================

[무엇을]
  - 서버: POST /api/extract. base64 + mediaType 받아 Claude
    (claude-sonnet-4-6) vision 호출. 시스템 프롬프트로 JSON 강제.
  - 클라: Today 페이지에서 카메라/파일 → 미리보기 → 표현 추출하기.
    이미지를 canvas 로 리사이즈(최대 1600px, target 3MB) + JPEG 재인코딩
    → base64 → API.
  - 데이터 모델: Session(주제·이미지·표현 ID 목록) + Expression
    (영어·한국어·유형·교재 예문).
  - localStorage 에 구조화 데이터, IndexedDB 에 이미지 Blob.

[왜]
  - Claude vision 은 5MB 한도가 있고, 폰 카메라 사진은 그대로 보내면
    base64 후 7~10MB 라 사전 압축이 필수.
  - 이미지는 작지 않으니 localStorage(약 5~10MB 한도)는 부적합.
    IndexedDB 가 이미지 저장의 표준 위치.
  - 데이터 모델은 ISO 주차(weekKey) 같은 키를 미리 박아두면 이후
    주간 통계·복습·캘린더가 다 거의 공짜로 따라온다.

[응용]
  - 멀티미디어 입력은 항상 "리사이즈 + 인코딩" 한 번 거치고 서버에 보낼 것.
    저장 한도와 API 한도 둘 다 보호된다.
  - 처음 데이터 모델을 짤 때 "나중에 어떤 통계/뷰가 나올까" 한 발 더
    내다보고 키를 미리 박아두면 리팩터링이 줄어든다.

================================================================
  STEP 4. Phase 2 — 표현 상세 + 발음(TTS)
================================================================

[무엇을]
  - useTTS / SpeakButton / SpeedToggle 공용 컴포넌트.
  - Web Speech API(speechSynthesis)로 영어 발음. en-US voice 우선,
    voiceschanged 이벤트 대기 + 2초 timeout fallback.
  - 속도 토글 0.75x / 1.0x, localStorage 영속화 + 인스턴스 간 broadcast.
  - 표현 상세 페이지 /expression/:id 신설.

[왜]
  - TTS 는 다행히 iOS Safari 도 지원 → 서버 호출 없이 즉시 발음 가능.
  - voice 목록 로딩이 비동기인 모바일 환경을 위해 대기·timeout 로직 필수.

[응용]
  - 외부 API 의존 없이 즉시 동작하는 클라 기능부터 붙이면 사용자
    feedback 의 첫 임팩트가 크다.
  - 사용자 설정(속도·테마 등) 영속화는 처음부터 localStorage 로.
    "한 번 끄면 다음에도 꺼져 있어야" 학습 도구가 부담 없이 켜진다.

================================================================
  STEP 5. Phase 3 — 유사 표현 + 내 문장 첨삭
================================================================

[무엇을]
  - 서버: POST /api/enrich (similar + extraExamples), POST
    /api/sentence-feedback ({isNatural, corrected, feedback, better}).
  - 클라: ExpressionDetail 에 SimilarSection + MySentencesSection 마운트.
  - 호출 결과를 Expression.similar / Expression.myExamples 에 캐싱 →
    같은 표현 다시 들어와도 API 재호출 안 함.
  - 서버 헬퍼 callClaudeForJSON() 로 모델 호출·코드펜스 제거·JSON 파싱
    공통 처리.

[왜]
  - 토큰 = 돈. "이미 받아둔 결과는 다시 부르지 않는다" 는 캐시 정책이
    1인용 앱에서도 큰 차이를 만든다.
  - 모델 응답은 종종 ```json ... ``` 같은 마크다운으로 포장된다 → 코드펜스
    스트리퍼는 거의 모든 LLM JSON 파이프라인의 필수 부품.

[응용]
  - LLM 응답을 JSON 으로 받는 모든 엔드포인트는 (1) 시스템 프롬프트로
    형식 강제 (2) 코드펜스 제거 (3) try/catch JSON.parse 의 3-step
    안전 처리. 더 강하게 묶고 싶으면 tool use 도 옵션.

================================================================
  STEP 6. Phase 4 — 주간 복습 + 플래시카드 (보고 말하기 / 타이핑)
================================================================

[무엇을]
  - Review 페이지: weekKey 별 그룹 + 진행률 바 + 암기율 %.
  - Flashcards 페이지 /review/:weekKey 신설.
  - 모드 토글: 보고 말하기 / 타이핑. 후자는 입력 후 자동 비교
    (case·whitespace·punctuation 정규화).
  - "암기 완료 / 다시" 버튼이 Expression.memorized + lastReviewed 토글.

[왜]
  - 비교 함수가 너무 빡빡하면 "정답인데 오답" 사용자 경험이 망가진다.
    Whisper 가 문장 끝에 마침표를 자동으로 붙이는 패턴도 잡아줘야 한다.
  - 진행률 / 암기율은 store 함수로 한 번 계산 → Review / Home / 캘린더에서
    재사용.

[응용]
  - 사용자 입력 vs 정답 비교는 항상 "정규화 함수" 한 곳에 모아라.
    버그 났을 때 거기만 고치면 모든 화면이 같이 좋아진다.

================================================================
  STEP 7. 모바일 마이크 입력의 험난한 여정
================================================================

[무엇을]
  - 처음엔 Web Speech API SpeechRecognition 으로 시도.
    데스크톱 Chrome 에서는 잘 됨. 안드로이드 Chrome 도 됨.
  - 아이폰 Chrome / Safari 모두 SR API 자체를 노출하지 않음 → 실패.
    (iOS 의 모든 브라우저가 WebKit 강제, 그 WebKit 이 SR 을 미구현)
  - 방향 전환: MediaRecorder 로 오디오 녹음 → 서버 /api/transcribe
    → OpenAI Whisper API → 텍스트 반환.
  - 서버 .env 에 OPENAI_API_KEY 추가, openai SDK 설치, toFile() 헬퍼.
  - 클라 useWhisperRecorder 훅 (useSpeechRecognition 과 같은 인터페이스).
  - Flashcards / Conversation 모두 같은 훅을 사용 → 동일한 UX.

[왜]
  - 브라우저 기반 STT 는 매력적이지만 iOS 는 사실상 막혀 있다.
    "PWA 인데 아이폰에서 마이크 안 됨" 은 1인용 학습 앱에선 치명적.
  - MediaRecorder 는 모든 모바일 브라우저에서 동작 → 서버에 보내
    Whisper 가 텍스트로 바꿔주는 라운드트립이 가장 호환성 좋음.

[응용]
  - 모바일 음성 기능은 처음부터 "녹음 → 서버 STT" 로 설계.
    크로스플랫폼 일관성·음성 정확도 둘 다 우위.
  - 한 번 만든 훅의 인터페이스(supported/listening/transcript/error/
    start/stop/reset)를 다른 구현에도 동일하게 유지하면 UI 코드 0줄
    수정으로 백엔드 교체 가능.

================================================================
  STEP 8. Phase 5 — 회화 채팅 (+ paraphrases)
================================================================

[무엇을]
  - 서버: POST /api/conversation 가 {topic, expressions, history} 받아
    {reply, paraphrases, nextQuestion} 반환.
  - 시스템 프롬프트: 친절한 코치, 한 번에 한 영어 질문, 답이 어색하면
    영어 제안, 그리고 paraphrases 로 같은 의미 다른 표현 2~3개.
  - 클라: Conversation 페이지. 세션 드롭다운 + 메시지 영역 + sticky 입력.
    iOS 키보드를 고려해 `h-[calc(100dvh-7.5rem)]` 로 동적 높이.
  - 어시스턴트 메시지는 reply 버블 + paraphrases 카드(노랑) + 다음 질문
    버블 3-blok 으로 분리.
  - 입력 textarea 옆에 마이크 버튼(useWhisperRecorder) → 음성으로도 답변.

[왜]
  - 학습 효율은 "같은 의미의 다른 표현 N가지" 를 자연스럽게 보여줘야
    크게 올라간다. 사용자가 요청해서 paraphrases 필드 추가했고, 이후
    모든 대화에서 가장 만족도 높은 UX 가 됨.
  - 모바일 chat UI 는 hidden 위치 함정이 많다. sticky bottom-0 은
    부모 스크롤 컨테이너에 가려 사라진다 → 페이지 자체를 flex column
    + 메시지 영역 내부 스크롤 + 입력 영역 flex item 으로 재구성.

[응용]
  - 모바일 chat 만들 땐 무조건 100dvh 베이스 + 페이지 내부 스크롤.
    100vh 는 모바일에서 keyboard / address bar 변화에 못 반응함.
  - 멀티턴 LLM API 는 실패 시 클라에서 낙관적 user 메시지를 롤백하지
    않으면 다음 호출이 "user 두 번 연속" 으로 또 실패한다.

================================================================
  STEP 9. claude-sonnet-4-6 의 JSON 강제 — 깊은 흙구덩이
================================================================

[무엇을]
  - 회화에서 모델이 시스템 프롬프트를 무시하고 평문으로 응답 → JSON
    파싱 실패. raw 로그를 보고 원인 파악.
  - 1차 시도: Anthropic 권장 "assistant prefill `{`" 기법 적용.
    → claude-sonnet-4-6 가 "This model does not support assistant
    message prefill" 400 으로 거부.
  - 2차 시도(채택): 마지막 user 메시지 끝에 "JSON 만 출력" 리마인더 자동
    추가 + parseJSONLoose 가 응답에서 balanced `{...}` 블록 추출.
  - 같은 사례 재호출 → 2.6s 만에 깔끔한 JSON 응답.

[왜]
  - 모델 버전마다 지원 기능이 다르다. 공식 가이드 라인의 패턴이라도
    실제 API 가 받아주지 않을 수 있어 항상 raw 응답 + 에러 메시지를
    살피는 습관이 필요.
  - "JSON 리마인더 + 안전 파싱" 조합은 어느 모델이든 동작하는 최저
    공통 분모.

[응용]
  - LLM 으로 구조화 응답 받는 모든 코드는 다음을 기본:
      1. 시스템 프롬프트에 스키마 명시
      2. 마지막 사용자 메시지에 형식 재확인 리마인더
      3. 응답에서 balanced `{...}` 추출하는 파서
      4. 그래도 안 되면 모델의 tool use 로 escalate

  STEP 10. Phase 6 — 영작 + 첨삭

[무엇을]

    safe-area-inset-bottom 대응.

[왜]
  - "마감 작업" 의 핵심은 사용자가 매일 켜는 동기 부여다. streak 와
    캘린더 점이 그 역할.
  - 백업/복원은 1인용 앱에서 데이터 신뢰성을 담보하는 거의 유일한 수단.
    DB 없이 브라우저 저장만 쓴다면 무조건 만들어야 한다.

[응용]
  - 다크모드 일괄 변환은 정규식 + 스크립트가 답. 한 파일씩 손으로 고치면
    실수가 누적된다.
  - PWA + localStorage/IndexedDB 만으로 운영할 거면 JSON export/import
    는 옵션이 아니라 필수.

================================================================
  STEP 12. 배포 — GitHub → Render(서버) + Vercel(클라이언트)
================================================================

[무엇을]
  - git init + 초기 커밋(50+ 파일). .env 는 .gitignore 로 차단.
  - GitHub private repo 만들고 push.
  - Render: render.yaml Blueprint 로 web service 1개 자동 생성. env 에
    ANTHROPIC_API_KEY / OPENAI_API_KEY / (선택) ALLOWED_ORIGIN.
  - Vercel: 같은 repo 를 Vite preset 으로 import. VITE_API_BASE_URL 에
    Render URL.

[중간에 잡힌 함정 3가지]
  1) render.yaml 은 반드시 **repo 루트**에 있어야 한다. server/render.yaml
     은 Render Blueprint 가 인식 못 한다.
  2) Vercel 의 "Root Directory" 설정이 일부 플랜 UI 에서 숨어 있다.
     루트 vercel.json 에 installCommand / buildCommand / outputDirectory
     를 직접 박아 우회.
  3) Vite 의 VITE_* env 는 **빌드 시점**에 JS 번들에 박힌다. Vercel
     환경변수만 추가하고 끝나는 게 아니라 반드시 "Redeploy(캐시 끄고)"
     까지 해야 새 번들이 만들어진다.

[왜]
  - Render 무료 플랜: 15분 idle 후 cold start 가 ~30초. 1인용엔 무난.
  - Vercel: 정적 호스팅 + 자동 git 연동 + 무료. 모바일 PWA 배포의
    표준 선택.

[응용]
  - 풀스택 1인 앱 무료 배포 조합: GitHub(코드) + Render(API/서버) +
    Vercel(클라/PWA) = 충분.
  - 첫 배포에서 90% 사람들이 빌드 디렉터리·환경변수 baking 한 번씩
    헛걸린다. 빌드 로그를 항상 끝까지 읽어라.
